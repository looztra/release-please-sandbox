# Aligning the release tag with the staging-deployed commit

## Target deployment workflow

The goal of the study is to support the following workflow:

1. Each time a commit is pushed to `main`, artifacts (e.g. a Docker image) are
   built and deployed to a **staging** environment, keyed by the commit SHA.
2. When a release is validated (the release-please PR is approved and merged),
   the release tag must point to the **same commit that was deployed to
   staging**, i.e. the last commit before the release PR merge commit.
3. A dedicated workflow (outside this repository) reacts to the tag and
   **retags the exact Docker image** that was deployed to staging with the
   release version.
4. Production is deployed with that retagged image. Staging and production run
   the **exact same image** (same `sha256sum`; the Docker tags differ, which
   is acceptable).

In this context, having `CHANGELOG.md` and `version.txt` out of sync with the
tagged tree is explicitly **not an issue**.

As established in
[release-please default behavior](release-please-default-behavior.md), the
commit to tag is always the **first parent of the release PR merge commit**.

## Options considered

### Option 1: keep the default, adjust consumers

Keep release-please untouched and have downstream jobs use
`${{ steps.release.outputs.sha }}^` to reference the staging commit.

Rejected: the retag workflow lives in another repository and is triggered by
the tag itself, so the tag must point to the right commit.

### Option 2: re-point the tag right after creation

Let release-please create the tag and the release, then force-move the tag
ref to the parent commit with an extra workflow step.

Rejected for this use case: the tag briefly exists at the wrong SHA before
being force-updated. An external workflow triggered by the tag can fire twice,
or once with the wrong commit. That race window is unacceptable when the tag
drives a production retag.

### Option 3: `skip-github-release` plus a custom publish job (selected)

Configure the action with `skip-github-release: true` so it only manages the
release PR. Add a dedicated job, triggered when the release PR is merged, that
creates the tag and the GitHub release in a **single `createRelease` API
call**, already pointing at the staging-deployed commit.

Selected because:

- The tag is created exactly once, at the right SHA: no race window for
  downstream consumers.
- The release notes can reuse the changelog content generated by
  release-please (it is embedded in the release PR body).
- Release-please keeps computing the next version correctly as long as the tag
  naming (`vX.Y.Z`) matches what it would have produced.

The detailed design is in
[proposed release workflow](release-workflow-proposal.md).

### Option 4: switch tools

Tools like semantic-release natively tag the actual commit (no release PR,
changelog kept in the release notes). Rejected: a much bigger change than
needed, and the release-PR-based validation flow ("a release is validated by
approving the release PR") is precisely what we want to keep.
